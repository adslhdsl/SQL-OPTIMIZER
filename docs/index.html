01에이전트(Agent)란 무엇인가?

단순 AI 호출과 에이전트의 핵심 차이를 이해합니다.

🤖 단순 AI 호출 (1-shot)

  • 사용자가 질문을 입력
  • LLM이 한 번 답변을 생성
  • 추가 정보 수집 불가
  • 답변 품질이 입력에 전적으로 의존
  • 외부 시스템과 상호작용 불가

🧠 에이전트 (Agent)

  • 목표만 주어지면 스스로 계획 수립
  • 필요한 정보를 도구(Tool)를 통해 직접 수집
  • 수집한 정보를 바탕으로 추가 추론
  • 외부 시스템 (DB, API, 파일 등) 과 상호작용
  • 충분한 정보가 모이면 최종 답변 생성

핵심 개념: 에이전트는 LLM + 도구 + 루프(반복)의 조합입니다. LLM이 단순히 답을 생성하는 것을 넘어, "지금 무엇을 더 알아야 하는가?"를 스스로 판단하고 도구를 호출하여 정보를 수집한 뒤, 그 결과를 바탕으로 다시 추론합니다.

🔄

반복(Loop)

LLM은 도구가 필요하지 않다고 판단할 때까지 반복적으로 도구를 호출하고 추론합니다. SQL Optimizer는 최대 10회 반복합니다.

🛠️

도구 사용(Tool Use)

LLM이 직접 실행할 수 없는 작업 (DB 조회, API 호출 등)을 코드로 구현된 함수를 통해 수행합니다.

💬

대화 이력(Context)

이전 도구 호출과 그 결과가 모두 누적되어 다음 LLM 호출에 전달됩니다. 에이전트의 "기억"입니다.

🎯

자율 판단(Autonomy)

어떤 도구를 어떤 순서로 호출할지 LLM이 스스로 결정합니다. 코드에는 하드코딩되어 있지 않습니다.

02프로젝트 개요

SQL Optimizer Agent는 무엇을 하는 프로젝트인가?

🎯

목적

사용자가 제공한 SQL 쿼리와 실행 계획(EXPLAIN)을 AI가 분석하여, 인덱스 추가·쿼리 재작성 등 구체적인 최적화 방안을 한국어로 제안합니다.

🔐

보안 설계

실제 데이터는 절대 조회하지 않습니다. 스키마, 인덱스, 행 개수 등 메타데이터만 수집합니다. 실행 계획은 사람이 직접 실행해서 붙여넣습니다.

🧩

구성

agent.py (메인 에이전트) + mcp_server.py (Claude Code 연동) + setup_db.py (실습용 샘플 DB)

📚

학습 목적

이 프로젝트는 에이전트의 핵심 패턴인 도구 호출 루프, 대화 이력 관리, 자율 판단을 명확하게 보여주기 위해 설계되었습니다.

파일 구조

sql-optimizer/
├── agent.py        # 메인 에이전트 — Gemini 호출 + 도구 루프
├── mcp_server.py   # MCP 서버 — Claude Code 연동용
├── setup_db.py     # 샘플 DB 생성 (인덱스 없는 의도적 비효율 스키마)
├── sample.db       # SQLite DB — customers(1000건) + products(200건) + orders(10000건)
├── requirements.txt # google-genai, mcp
└── .env            # GEMINI_API_KEY

각 단계를 클릭하면 상세 설명을 볼 수 있습니다.

① 초기 컨텍스트 구성

에이전트가 시작될 때 첫 번째로 하는 일입니다. 사용자가 제공한 SQL 쿼리와 EXPLAIN 결과를 Gemini에게 전달할 형태로 포장합니다.

# run_agent() 시작 부분
contents = [
  types.Content(
    role="user",
    parts=[types.Part(text=f"""
다음 SQL 쿼리와 실행 계획을 분석해주세요.

SQL: {sql_query}
실행 계획: {explain_result}
    """)]
  )
]

포인트: contents는 Python 리스트입니다. 대화가 진행될수록 도구 호출과 결과가 계속 이 리스트에 추가됩니다. 이것이 에이전트의 "작업 기억(Working Memory)"입니다.

② Gemini API 호출

현재까지 누적된 대화 이력과 사용 가능한 도구 목록을 Gemini에게 전달합니다. Gemini는 이 정보를 바탕으로 다음 행동을 결정합니다.

config = types.GenerateContentConfig(
  tools=TOOL_FUNCTIONS,  # 4개 함수 자동 변환
  system_instruction=SYSTEM_PROMPT
)

response = client.models.generate_content(
  model="gemini-2.5-flash",
  contents=contents,   # 누적된 대화 이력
  config=config,
)

포인트: tools=TOOL_FUNCTIONS에 Python 함수를 직접 넘기면, Google GenAI SDK가 함수의 이름, docstring, 타입 힌트를 읽어 JSON 스키마로 자동 변환합니다.

③ 응답 분석 — 분기점

Gemini의 응답에서 function_call이 있는지 확인합니다. 이것이 루프를 계속할지, 최종 답변을 반환할지 결정하는 핵심 분기입니다.

candidate = response.candidates[0]
contents.append(candidate.content)  # 항상 누적

# function_call 추출
fn_calls = [
  p.function_call
  for p in candidate.content.parts
  if p.function_call
]

if not fn_calls:
  # 도구 필요 없음 → 최종 답변
  return "".join(
    p.text for p in candidate.content.parts
    if p.text
  )

④ 도구 실행

Gemini가 요청한 함수를 TOOL_MAP을 통해 찾아서 실제로 실행합니다.

# TOOL_MAP: 함수 이름 → 실제 함수 객체
TOOL_MAP = {
  fn.__name__: fn
  for fn in TOOL_FUNCTIONS
}

fn_response_parts = []
for fc in fn_calls:
  fn = TOOL_MAP.get(fc.name)
  result = fn(**dict(fc.args))

  fn_response_parts.append(
    types.Part.from_function_response(
      name=fc.name,
      response={"result": result}
    )
  )

포인트: 예를 들어 Gemini가 get_schema("orders")를 요청하면, TOOL_MAP["get_schema"]("orders")가 실행되어 SQLite에서 orders 테이블의 컬럼 구조를 가져옵니다.

⑤ 결과 누적

도구 실행 결과를 contents에 추가합니다. 다음 루프에서 Gemini는 이 결과를 참조하여 추가 추론을 수행합니다.

# 도구 결과를 대화 이력에 추가
contents.append(
  types.Content(
    role="user",
    parts=fn_response_parts
  )
)
# → 다음 루프에서 Gemini가 이 결과를 참조

누적되는 contents 구조:
[사용자 질문] → [Gemini 응답 + function_call] → [도구 결과] → [Gemini 응답 + function_call] → [도구 결과] → ... → [최종 답변]

⑥ 최종 답변 생성

Gemini가 충분한 정보를 수집했다고 판단하면, 더 이상 function_call을 포함하지 않고 텍스트 답변만 생성합니다. 이때 루프가 종료됩니다.

# Gemini가 반환한 최종 텍스트 수집
if not fn_calls:
  return "".join(
    p.text
    for p in candidate.content.parts
    if p.text
  )

최종 출력 예시:

## 현재 쿼리의 문제점
1. orders 테이블 전체 스캔(SCAN) 발생
2. customers 테이블도 인덱스 없이 조회됨

## 제안하는 인덱스
CREATE INDEX idx_orders_customer_status
  ON orders(customer_id, status);

CREATE INDEX idx_customers_region
  ON customers(region);

## 예상 성능 개선

05도구(Tool) 시스템

에이전트가 사용할 수 있는 4개의 도구와 각각의 역할

🗂️ get_all_tables()
데이터베이스에 존재하는 모든 테이블 이름을 조회합니다. Gemini가 가장 먼저 호출하는 도구입니다.
SELECT name FROM sqlite_master WHERE type='table'
📋 get_schema(table_name)
특정 테이블의 컬럼 구조(이름, 타입, NOT NULL, 기본값)를 조회합니다.
PRAGMA table_info(table_name)
🔍 get_indexes(table_name)
특정 테이블에 존재하는 인덱스 목록을 조회합니다. 최적화 제안의 핵심 정보입니다.
PRAGMA index_list(table_name)
📊 get_table_stats(table_name)
테이블의 행 개수를 조회합니다. 대용량 테이블 여부를 판단하는 데 활용됩니다.
SELECT COUNT(*) FROM table_name

도구 등록 방식 — Python 함수 → JSON 스키마 자동 변환

코드에서 이렇게 등록하면 →

def get_schema(table_name: str) -> str:
  """특정 테이블의 컬럼 구조를 반환합니다."""
  ...

TOOL_FUNCTIONS = [
  get_all_tables,
  get_schema,
  get_indexes,
  get_table_stats,
]

config = types.GenerateContentConfig(
  tools=TOOL_FUNCTIONS
)

SDK가 자동으로 이렇게 변환 →

{
  "name": "get_schema",
  "description": "특정 테이블의
    컬럼 구조를 반환합니다.",
  "parameters": {
    "type": "object",
    "properties": {
      "table_name": {
        "type": "string"
      }
    },
    "required": ["table_name"]
  }
}

중요: 함수의 docstring이 Gemini에게 전달되는 도구 설명이 됩니다. Gemini는 이 설명을 읽고 언제 이 도구를 사용할지 스스로 판단합니다. 좋은 docstring이 에이전트의 품질을 결정합니다.

06동작 예시 — 단계별 워크스루

실제 쿼리를 입력했을 때 에이전트 내부에서 일어나는 일들을 순서대로 추적합니다.

입력 쿼리 (최적화 전)

SELECT c.name, COUNT(o.id) AS order_count
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE c.region = 'Seoul'
  AND o.status = 'pending'
GROUP BY c.id
ORDER BY order_count DESC
LIMIT 10;

EXPLAIN QUERY PLAN 결과 (사람이 직접 실행)

QUERY PLAN
|--SCAN orders          ← 10,000건 전체 스캔!
|--SEARCH customers USING INTEGER PRIMARY KEY (rowid=?)
`--USE TEMP B-TREE FOR GROUP BY
1

사용자 입력 → run_agent() 호출

SQL 쿼리와 EXPLAIN 결과를 run_agent()에 전달합니다. 이 두 값이 contents 리스트의 첫 번째 항목이 됩니다.

result = run_agent(
  sql_query=test_query,
  explain_result=test_explain
)

루프 1회 — Gemini: "테이블 목록이 필요해"

Gemini는 쿼리에 등장하는 테이블이 DB에 실제로 존재하는지 확인하려 합니다. get_all_tables를 호출합니다.

# Gemini → function_call
{
  "name": "get_all_tables",
  "args": {}
}

# 실행 결과
["customers", "products", "orders"]
3

루프 2회 — Gemini: "orders 스키마가 필요해"

EXPLAIN에서 SCAN orders가 있으니 orders 테이블의 컬럼 구조와 인덱스를 확인합니다.

# Gemini → function_call (병렬 호출 가능)
{"name": "get_schema", "args": {"table_name": "orders"}}
{"name": "get_indexes", "args": {"table_name": "orders"}}

# get_schema 결과
[{cid:0, name:"id", type:"INTEGER", pk:1},
 {cid:1, name:"customer_id", type:"INTEGER"},
 {cid:2, name:"product_id", type:"INTEGER"},
 {cid:3, name:"quantity", type:"INTEGER"},
 {cid:4, name:"status", type:"TEXT"},    ← WHERE 조건
 {cid:5, name:"ordered_at", type:"TEXT"}]

# get_indexes 결과
[]  ← 인덱스 없음! 문제 발견
4

루프 3회 — Gemini: "customers도 확인해야겠어"

customers 테이블의 region 컬럼도 WHERE 조건에 포함되어 있으므로 해당 인덱스도 확인합니다.

# Gemini → function_call
{"name": "get_schema", "args": {"table_name": "customers"}}
{"name": "get_indexes", "args": {"table_name": "customers"}}
{"name": "get_table_stats", "args": {"table_name": "orders"}}

# 통계 결과
{"table": "orders", "row_count": 10000}  ← 대용량 확인
5

루프 4회 — Gemini: "충분해, 답변 작성"

필요한 정보가 모두 수집되었습니다. function_call 없이 텍스트 답변만 반환합니다. 루프가 종료됩니다.

# Gemini 응답에 function_call 없음 → return

최종 출력 — 한국어 최적화 제안

Gemini가 생성한 구체적인 최적화 방안입니다.

## 1. 현재 쿼리의 문제점

orders 테이블(10,000건)에서 인덱스 없이 전체 스캔(SCAN)이
발생하고 있습니다. status = 'pending' 조건과
customer_id = c.id JOIN 조건에 인덱스가 없어 매 조회마다
10,000건을 모두 읽습니다.

## 2. 제안하는 인덱스

-- orders: 복합 인덱스 (status 필터 + customer_id JOIN)
CREATE INDEX idx_orders_status_customer
  ON orders(status, customer_id);

-- customers: region 필터 인덱스
CREATE INDEX idx_customers_region
  ON customers(region);

## 3. 예상 성능 개선

status = 'pending' 비율이 ~20%라면:
  현재: 10,000건 스캔 → 개선 후: ~2,000건으로 축소
  JOIN도 인덱스로 처리 → 추가 80% 향상 기대
  전체 약 50~100배 성능 개선 예상
7

contents 리스트 최종 구조

루프가 끝난 뒤 contents에 누적된 대화 이력입니다. 이것이 에이전트의 전체 "사고 과정"입니다.

contents = [
  # 1. 사용자 초기 입력
  Content(role="user", parts=["SQL + EXPLAIN"]),
  # 2. Gemini: get_all_tables 호출
  Content(role="model", parts=[function_call(get_all_tables)]),
  # 3. 도구 결과
  Content(role="user", parts=[function_response(["customers","orders",...])]),
  # 4. Gemini: get_schema + get_indexes 호출
  Content(role="model", parts=[function_call(...), function_call(...)]),
  # 5~8. 추가 도구 호출 및 결과...
  # 9. Gemini 최종 텍스트 답변
  Content(role="model", parts=["## 현재 쿼리의 문제점..."]),
]
